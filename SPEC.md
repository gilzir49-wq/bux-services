## 1. מה הכלי עושה (תמצית פונקציונלית)

כלי פנימי לצוות המשרד של CrossFit BUX. שני דברים מרכזיים, מאוחדים:

1. **רישום פעילות** — כל אחד רושם מה עשה (סוג פעילות, מתי, כמה זמן, עם מי).
   משמש למעקב/דוח חודשי.
2. **לוח המשרד** — לוח תפוסה משותף של **מרחב אחד** (המשרד). כל תפיסה חוסמת אותו
   לכולם, כדי שלא ייקבעו שניים על אותו זמן.

הכל **פתוח וחופשי**: נכנסים עם שם בלבד (בלי קודים/סיסמאות), וכולם רואים את הכל
ויכולים לערוך הכל.

## 2. משתמשים והרשאות (מודל פתוח)

- **כניסה בשם בלבד** — בוחרים שם מרשימה → נכנסים. אין קוד אישי, אין קוד מנהל.

## 3. סוגי פעילות (Activity types)

```
ישיבת צוות · צילומי תוכן · ייעוצי תזונה · טיפולי ספורטרפיה · עבודה משרדית
```

- לכל סוג יש **צבע**.
- **סוגים ללא "עם מי"** (אין מטופל ספציפי): `ישיבת צוות`, `צילומי תוכן`, `עבודה משרדית`.
  כשבוחרים אחד מהם, שדה "עם מי (שם המתאמן/ת)" **נעלם** והופך ללא-חובה.
- בייעוצי תזונה וטיפולים — שדה "עם מי" מופיע.

## 4. מודל נתונים

### providers (הצוות)
```
id → { name, role, active, rate (ריק, לעתיד), color, email }
```

### sessions (רישום פעילות שקרתה)
```
key → { provider_id, date (YYYY-MM-DD), start_time (HH:MM),
        duration_minutes, service_type, client_name, note, created_at (ISO) }
```

### bookings (תפיסות עתידיות של המשרד בלוח)
```
key → { provider_id, kind: "once"|"weekly",
        date? (ל-once),  weekday? (0=ראשון..6=שבת, ל-weekly),
        start (HH:MM), end (HH:MM), note, created_at }
```

**חישוב תשלום הושאר ריק בכוונה** — פונקציית `calcPayment(session, provider)`
מחזירה `null`. לכל נותן שירות סידור תשלום שונה שיוגדר בעתיד (שדה `rate` שמור).
כרגע הכלי רק סופר ומרכז, בלי חישוב כספי.

## 5. מסך רישום פעילות

- טופס: תאריך (ברירת מחדל היום), שעת התחלה (ברירת מחדל עכשיו), משך בדקות,
  סוג פעילות (ברירת מחדל לפי התפקיד של המשתמש), "עם מי" (מוסתר בסוגים ללא מטופל),
  הערה. כפתור "שלח רישום".
- מתחת: **רשימת כל הרישומים** של כל הצוות (מהחדש לישן), עם שם האדם על כל רישום,
  ואפשרות עריכה/מחיקה לכל אחד (פתוח).

---

## 6. לוח המשרד (הלב של הכלי)

- **מרחב אחד משותף** — כל תפיסה חוסמת אותו לכולם.
- **ניווט שבועי**: חיצים ‹ › + כפתור "היום". רצועת 7 ימים (א׳–ש׳ עם תאריכים),
  היום מסומן, בכל יום badge עם מספר התפיסות.
- **תצוגת יום נבחר**: רשימת התפיסות של אותו יום, ממויינות לפי שעה, בצבע האדם.
- **הוספת תפיסה**: בוחרים אדם (ברירת מחדל = המשתמש הנוכחי), תאריך, שעה מ-/עד,
  הערה, ו-checkbox **"חוזר כל שבוע"**:
  - מסומן → נשמר כ-`weekly` עם `weekday`, ומופיע בכל שבוע.
  - לא מסומן → נשמר כ-`once` לתאריך הבודד.
- **זיהוי חפיפות ⚠️**: שתי תפיסות שחופפות בזמן מסומנות באדום עם "⚠️ חפיפה".
- עריכה/מחיקה פתוחות לכולם.

### אינטגרציה בין רישום ללוח (חשוב!)
הלוח מציג **גם את הרישומים שנרשמו** (sessions) כמשבצות תפוסות, לא רק תפיסות ידניות:
- כל `session` באותו תאריך מוצג בלוח עם `start_time` → `start_time + duration`.
- מתויג **📝 מפגש** (מקורו ברישום), לעומת **🗓️ תפיסה** (נקבע ידנית בלוח) /
  **↻ כל שבוע** (חוזר).
- שני המקורות נספרים יחד ונבדקים יחד לחפיפות.

הפונקציה המרכזית: `slotsForDate(iso)` — ממזגת `bookings` + `sessions` ליום נתון.

## 7. סיכום / דוח (פתוח לכולם)

- בחירת חודש / סינון לפי אדם / סוג / טווח תאריכים.
- טבלה מרכזת: לכל אדם — מספר פעילויות, סה"כ שעות, פירוק לפי סוג. שורת סה"כ.
- לחיצה על שורה → כל הפעילויות המפורטות של אותו אדם.
- **ייצוא ל-CSV** עם BOM (עברית תקינה באקסל).

## 8. התראות מייל + הוספה ליומן

כשנשמרת **תפיסת משרד חדשה** (לא עריכה, לא רישום), קופץ חלון:
**"📧 לשלוח התראה לצוות?"**

- מציג סיכום (מתי / מי / הערה) ואת רשימת הנמענים.
- כפתור **"שלח מייל לצוות"** = קישור `mailto:` שנפתח באפליקציית המייל של הטלפון,
  ממוען מראש **לכל שאר הצוות** (כולם חוץ ממי שתפס) שיש להם מייל, עם נושא + גוף
  מוכנים. השולח רק לוחץ "שלח".
- בגוף המייל: קישור **"➕ הוספה ליומן"** = Google Calendar template URL
  (`calendar.google.com/calendar/render?action=TEMPLATE&dates=…&ctz=Asia/Jerusalem`).
  לתפיסה חוזרת נוסף `recur=RRULE:FREQ=WEEKLY`.

**בלי שרת ובלי איסוף מיילים מרכזי:** כל אחד מזין את **המייל שלו** פעם אחת —
בכניסה הראשונה בלי מייל האפליקציה שואלת אותו (נזכר ב-`localStorage`), ותמיד אפשר
לשנות דרך כפתור ✉️ בכותרת. הנמענים מתמלאים אוטומטית בכל שליחה מרגע שיש להם מייל.

> הערה למטמיע: בגלל שזה אתר סטטי, השליחה היא דרך `mailto` (השולח לוחץ "שלח").
> באפליקציה עם backend אפשר לשדרג לשליחה **אוטומטית לגמרי** (בלי ללחוץ שלח) +
> **הזמנת יומן אמיתית** (קובץ `.ics` עם `METHOD:REQUEST` שמופיע ישר ביומן),
> ע"י שליחת מייל מהשרת. ראו סעיף 10.

## 9. תשתית טכנית (המימוש הנוכחי)

- **Frontend:** קובץ `index.html` יחיד, Vanilla JS + CSS (בלי build), RTL, PWA
  (manifest + service worker + אייקונים 192/512).
- **Backend:** Firebase **Realtime Database** (פרויקט `command-center-gal`,
  region `europe-west1`), ענף `bux-services`. גישה דרך REST (`fetch`):
  - `GET  /bux-services/providers.json`
  - `GET/POST  /bux-services/sessions.json` · `PATCH/DELETE /bux-services/sessions/<key>.json`
  - `GET/POST  /bux-services/bookings.json` · `PUT/DELETE /bux-services/bookings/<key>.json`
  - `PUT  /bux-services/providers/<id>/email.json`
  - רשומות נפרדות (POST push keys) כדי שכמה אנשים יכתבו במקביל בלי לדרוס.
  - חוקי אבטחה: הענף `bux-services` פתוח לקריאה/כתיבה; שאר הפרויקט חסום.
- **Hosting:** GitHub Pages (חינם).

## 10. הנחיות להטמעה באפליקציית הניהול הקיימת

המטרה: לאחד את הכלי לתוך אפליקציית הניהול (למשל `bux-manager`, Next.js 14 +
Supabase) במקום כלי נפרד.

**מה מעבירים כמודול "משרד / תפעול":**
1. **טבלאות** (מתרגמים מ-RTDB ל-Supabase Postgres):
   - `providers` → אפשר להשתמש במשתמשי/צוות הקיימים באפליקציה במקום טבלה נפרדת;
     מוסיפים שדות `color`, `email`, `role` אם חסרים.
   - `office_sessions` (= sessions): `provider_id, date, start_time,
     duration_minutes, service_type, client_name, note, created_at`.
   - `office_bookings` (= bookings): `provider_id, kind, date, weekday,
     start, end, note, created_at`.
2. **אימות** — משתמשים ב-auth הקיים של האפליקציה (הזהות כבר ידועה), במקום
   ה"כניסה בשם" הפתוחה. סוגי הפעילות והצבעים נשארים.
3. **מסכים** — להוסיף שלושה tabs/מסכים: רישום פעילות, לוח המשרד, סיכום. או לשלב
   את לוח המשרד בתוך לוח/יומן קיים באפליקציה.
4. **אינטגרציה** — לשמור על `slotsForDate` (מיזוג sessions+bookings ללוח)
   ועל זיהוי החפיפות.
5. **התראות** — עם backend (Next.js API route / Supabase Edge Function) אפשר:
   - לשלוח מייל **אוטומטית** בעת יצירת תפיסה (למשל דרך Resend/SendGrid/SMTP).
   - לצרף **הזמנת יומן `.ics` אמיתית** (VEVENT + `METHOD:REQUEST`) שנכנסת ישר
     ליומני הנמענים — במקום קישור "הוסף ליומן" ידני.
   - להשתמש ברשימת המיילים של המשתמשים הקיימים (בלי לבקש שוב).
6. **תשלום** — לממש את `calcPayment()` לפי סידור לכל נותן שירות (שדה `rate`).

## 11. לוגיקות מרכזיות ששווה לשמר (שמות פונקציות במימוש הנוכחי)

- `slotsForDate(iso)` — מיזוג תפיסות + רישומים ליום, למניעת דאבל-בוקינג.
- `markConflicts(list)` — סימון חפיפות זמן.
- `toggleClientField()` — הסתרת "עם מי" בסוגים ללא מטופל.
- `offerNotify(booking)` — בניית ה-mailto + קישור היומן לפי התפיסה.
- `gcalUrl(...)` — בניית Google Calendar template URL (כולל RRULE לחוזר).
- `exportCSV()` — ייצוא עם BOM.
